# Development

Internal guide for working on `sdk-node`. End-user docs live in the [README](../README.md).

## Toolchain

- **[bun](https://bun.sh)** — package manager and task runner.
- **[biome](https://biomejs.dev)** — formatter + import organizer (see `biome.json`).
- **[tsc](https://www.typescriptlang.org)** — type-checking and the published build (`dist/`).
- **[vitest](https://vitest.dev)** — test runner.
- **[buf](https://buf.build/docs/installation)** + **`@bufbuild/protoc-gen-es`** (devDependency) — regenerate the vendored protobuf code. Only needed when updating protos.

## Protobuf types (vendored)

Protobuf types are **vendored**, not pulled from a Buf Schema Registry npm package: the generated
code is committed under `src/gen/` (compiled into `dist/gen/` and shipped), so a consumer's
`npm install` needs **no `@buf` registry entry** in their `.npmrc`. The `.proto` sources are
mirrored under `proto/` from a **pinned** BSR commit. See the `Makefile`:

| Command             | What it does                                                                          |
| ------------------- | ------------------------------------------------------------------------------------- |
| `make sync-protos`  | `buf export buf.build/pugsh/pug:$(PROTO_COMMIT)` (read-only) → `proto/` mirror, allowlisted to the SDK packages (incl. the server-only `shared/**`). Needs BSR network. |
| `make protos`       | `buf generate` (local `protoc-gen-es`) → `src/gen/`, then `make typed-events`. Offline. |
| `make typed-events` | Regenerate `src/well-known-events.generated.ts` from a throwaway compile of `src/gen`. |
| `make proto-latest` | Print the newest commit on the module's `main` branch (to bump `PROTO_COMMIT`).        |

To take a newer schema: run `make proto-latest`, set `PROTO_COMMIT` in the `Makefile`, then
`make sync-protos && make protos`, and review + commit the `proto/` + `src/gen/` diff. Builds and
CI never touch the BSR — they compile the committed snapshot.

## Setup

```bash
bun install   # or: npm install — no .npmrc / registry config needed
```

## Scripts

| Command              | What it does                                                       |
| -------------------- | ----------------------------------------------------------------- |
| `bun run build`      | `prebuild` (stamp `version.ts`) → `tsc` → `dist/`                  |
| `bun run watch`      | `tsc -w` incremental rebuilds                                     |
| `bun run test`       | `vitest run` (one-shot)                                           |
| `bun run test:watch` | `vitest` in watch mode                                            |
| `bun run lint`       | `biome check --write .` (lint + format + organize imports)       |
| `bun run format`     | `biome format --write .`                                          |

`prebuild` only stamps `src/version.ts` (`SDK_VERSION` from `package.json`, applied to every
event's auto-properties). The protobuf code and the well-known-event catalog are **not**
regenerated by `build` — they are committed and refreshed via `make protos` when protos change
(CI diffs them for drift). Then `tsc` emits `dist/` per `tsconfig.json` (declarations on,
`*.test.ts` excluded). Only `dist/`, `README.md`, and `LICENSE` are published (`files` in
`package.json`).

## Well-known events

`src/well-known-events.generated.ts` is **autogenerated — do not edit by hand.** Regenerate with
`make typed-events` (or the full `make protos`). The generator (`scripts/gen-well-known-events.mjs`):

- Introspects the generated `*_events_pb` modules — via a throwaway tsc compile of `src/gen` into
  `.codegen-tmp` (node can't import the `.ts` source directly) — and collects every
  `*PropertiesSchema` that carries a `(common.events.v1.kind)` option.
- **Filter policy:** includes events whose `(common.events.v1.platforms)` option contains
  `PLATFORM_SERVER` **or** is unset (platform-neutral). Events tagged only with non-server
  platforms (web/mobile/desktop) are excluded.
- Emits a frozen `wellKnownSchemas` map keyed by event kind.

At runtime, `track()` looks up the kind in this map: known kinds get their properties validated
against the proto schema (with int-vs-double and other scalar types preserved from the field
definition), and unknown kinds fall back to the loose-property heuristic.

## Source layout

| File                          | Responsibility                                                     |
| ----------------------------- | ------------------------------------------------------------------ |
| `pug.ts`                      | `Pug` client — ingestion (`track`/`identify`) + reads facade.      |
| `track.ts`                    | `toEvent` — builds/validates Event protos, maps JS → PropertyValue. |
| `batch.ts`                    | In-memory batching transport: buffer, flush, retry, dead-letter.   |
| `rpc.ts` / `api-transport.ts` | Connect clients + transport with the `x-api-key` interceptor.      |
| `transport.ts`                | Adapts the events client into the batch transport's `EventSink`.   |
| `errors.ts`                   | `PugError` + `toPugError` normalization.                           |
| `well-known-events.ts`        | Public `TrackFn` / `TrackOptions` types over the generated catalog. |
| `logger.ts`, `utils.ts`       | Console logger + constants (default endpoint/timeout).             |

## Testing notes

- Tests live next to the code as `*.test.ts` and run through real `protovalidate` (the validator
  is not mocked), so they exercise the actual validation contract.
- Network is never dialed in unit tests — the batch transport is driven through a fake
  `EventSink`, and client tests point `endpoint` at `http://localhost:1` and only exercise the
  client-side guard/validation paths that return before any send.
- The throw-free ingestion contract is asserted with `expect(...).not.toThrow()` plus
  console spies that confirm the failure was logged.

Before opening a PR: `bun run lint && bun run test && bun run build`.
